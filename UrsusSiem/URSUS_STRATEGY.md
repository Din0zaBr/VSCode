## Часть 1. Сокращённый MVP

### 1.1 Главный принцип

Текущий план — 14 спринтов, ~16 недель работы. Для команды из 1–2 человек это нереально по двум причинам:

1. **Никто не дойдёт до конца** в одиночку без обратной связи от клиентов. Через 8 недель кодинга без продаж — выгорание гарантировано.
2. **Рынок изменится за 16 недель.** KUMA, MaxPatrol, UserGate выпустят новые версии. Ты должен оказаться у первого клиента максимум через 8 недель — иначе теряешь окно.

**Новое правило:** v2.0 release = **8 недель**, после которого начинаются продажи. Всё, что не нужно для первой коммерческой инсталляции — в v2.1, v2.2, в backlog.

### 1.2 Критерий "must-have" для MVP

Фича попадает в MVP только если **без неё первый клиент не купит** или **без неё демо провалится**.

### 1.3 Новая roadmap — 8 недель до v2.0

**Цель v2.0:** «У МСБ за 30 минут есть работающий SIEM с ФСТЭК-отчётом и Telegram-алертами».

| Неделя | Sprint | Главная цель | DoD |
|---|---|---|---|
| 1 | **Foundation** | Storage abstraction + DuckDB + syslog UDP/TCP + Vector ingest | `docker run ursus-siem` принимает syslog, видны в UI |
| 2 | **Content & Alerts** | 20 YAML-сценариев + Telegram/Email/Webhook + русские шаблоны | «RDP brute force» симулирован → Telegram-алерт за 60 сек |
| 3 | **Audit & TI** | Audit-log + AbuseCH puller + bloom-filter IOC + базовый GeoIP по диапазонам | Событие с IP из URLhaus → critical incident автоматом |
| 4 | **OCSF lite** | OCSF mapping для 3 классов (auth, process, network) + расширение схемы DuckDB | `SELECT actor.user.name FROM logs` работает |
| 5 | **Single-binary + onboarding** | `make all-in-one` + `install.sh` + UI-wizard + demo-injector | Чистая Ubuntu → готовый URSUS за 5 минут |
| 6 | **Compliance** | Шаблоны Приказа №21, №31 + typst PDF + экспорт audit-log + документ "Самопроверка" | Кнопка "Сгенерировать отчёт" → PDF с пунктами |
| 7 | **Tests + docs** | Unit-tests ≥40% (core ≥60%) + e2e bash-скрипты + `docs.ursus-siem.ru` (6 разделов) | `make test` зелёный, docs живой |
| 8 | **Polish + первый клиент** | Bugfix недели 1–7, демо-сценарий, первая инсталляция у реального человека | Один живой пилот, не разработчик |

**После v2.0** — пауза на кодинг, **минимум 4 недели только продажи и feedback**. Только после первых 3 платящих пилотов возвращаешься к коду.

### 1.4 Что в v2.1 (после первых 5 клиентов)

В порядке приоритета:
1. **ClickHouse backend** — если хотя бы один клиент упирается в DuckDB
2. **MITRE ATT&CK heatmap** — для демо и отчётности
3. **Cloud puller для Yandex Cloud** — если запросят
4. **UEBA модуль** — если оператор скажет «хочу видеть аномалии по пользователям»
5. **License split / Pro-фичи** — когда у тебя есть платящий клиент готовый платить $200+/мес
6. **MaxMind GeoIP** — точный impossible-travel
7. **Backup/restore CLI** — когда `cp` перестанет хватать

### 1.5 Что в v2.2 (через 3–6 месяцев)

- **LLM** (Vikhr 7B) — wow-фактор для презентаций и Habr-статей
- **EDR-light агент** — после первого крупного клиента
- **AD/DB canaries** — расширение honeypot
- **Cloud puller'ы для AWS/Azure/M365** — если будут международные клиенты
- **Multi-tenant SaaS** — если выйдешь на MSSP-канал
- **Сертификация ФСТЭК** — отдельный коммерческий трек, 6–12 месяцев

### 1.6 Бюджет времени по неделям (одиночка)

8 недель × 40 часов = **320 часов кодинга**. Это очень мало. Расклад честно:

| Активность | Часов |
|---|---|
| Чистый кодинг (Rust + Go) | 180 |
| Документация (русский) | 40 |
| 20 SIGMA-правил с тестами | 30 |
| Compliance-шаблоны | 20 |
| Тесты (юнит + e2e) | 30 |
| DevOps (install.sh, Docker, systemd) | 20 |
| **Итого** | **320** |

Если не одиночка, а двое — раздели как Rust-core (один) + Go API/UI/content/docs (второй). Тогда реалистично за 8 недель.

### 1.7 Что не делать ни в коем случае

- ❌ **Не переписывать API gateway с Go на Rust+axum.** В §16 плана это правильно отмечено как «требует обсуждения» — ответ: оставить Go. Переписывание = 2 недели потерянных.
- ❌ **Не делать helm chart для Enterprise.** До первого Enterprise-клиента нет смысла.
- ❌ **Не строить multi-tenant SaaS.** До 10+ инсталляций один-владелец-один-сервер.
- ❌ **Не вкладываться в API-биллинг.** Биллинг = ручной счёт-фактура первые полгода.
- ❌ **Не открывать GitHub раньше времени.** OSS-релиз без docs и без 20 правил из коробки = плохое первое впечатление. Сначала всё доделать, потом релиз.

## Часть 3. Позиционирование против каждого конкурента

Главное правило позиционирования: **не пытайся быть лучше всех во всём.** Выбираешь одну характеристику, где ты сильнее, и делаешь её главным сообщением.

### 3.1 Главный позиционный тезис URSUS

> **«SIEM, который МСБ может купить, развернуть и эксплуатировать без отдельного ИБ-специалиста — с готовым отчётом по Приказу №21 ФСТЭК».**

Три глагола: **купить** (по цене), **развернуть** (за 5 минут), **эксплуатировать** (без аналитика). Четвёртая часть — **compliance** — закрывает главного покупателя в РФ.

Если фраза слишком длинная для маркетинга — сократить до:

> **«URSUS — SIEM для тех, у кого нет SOC».**

или

> **«30 секунд установка. 5 минут до первого алерта. 0 строк YAML».**

### 3.2 Против KUMA

#### Их сила
- Бренд «Касперский» — продаёт сам по себе ЛПР'ам, которые «слышали»
- Глубокая интеграция с Kaspersky Endpoint Security (KES)
- Своя Threat Intelligence (KL TI) с петабайтами данных
- Сертификация ФСТЭК уже есть
- ГосСОПКА-модуль готов

#### Их слабость для МСБ
- **Лицензирование по EPS — финансовый шок.** Минимальная лицензия 100 EPS уже от ~100K ₽/год, а реально нужно 500–1000 EPS даже для 30 хостов
- **Развёртывание сложное** — несколько компонентов, своя БД, требует инженера на 1–3 дня
- **8+ ГБ RAM минимум** — на VPS уже от 5K ₽/мес
- Архитектура заточена под средний+ бизнес — для МСБ overkill
- Привязка к экосистеме Касперского: ценность KUMA максимальна только если стоит KES

#### Тезис URSUS против KUMA
> **«Всё, что нужно МСБ от KUMA, без оплаты за бренд и без интеграции с антивирусом, который у вас не куплен».**

#### Где НЕ конкурировать
Если у клиента уже есть KES corporate-лицензия (тысячи хостов) — не пытайся забрать его. KUMA + KES — связка. Целься в клиентов, у которых **не Kaspersky** в стеке: те, кто стоит на ESET, Dr.Web, открытых решениях, либо ещё не выбрал AV.

#### Если ЛПР задаёт прямой вопрос «А чем вы лучше KUMA?»
**Ответ-template:**
> «KUMA — отличный продукт, но он для другого профиля. Минимальная стоимость владения KUMA на нашем сегменте в первый год — от 1.5 млн рублей с учётом инженера на внедрение. URSUS вы развернёте сами за 5 минут, получите готовые отчёты по Приказу 21 в первую неделю, и заплатите 0 за core. Если потом разрастётесь и захотите KUMA — данные через OCSF-формат экспортируются».

Ключевой ход: **похвалить KUMA** (не критиковать бренд Касперского, ЛПР могут обидеться) + **развести по сегментам** (мы для МСБ, они для среднего+).

### 3.3 Против Wazuh

#### Их сила
- 3000+ готовых правил из коробки
- Огромное мировое community
- 10+ лет зрелости
- Бесплатно полностью
- Хорошая документация (английская)
- HIDS-функционал (FIM, rootkit detection) встроен

#### Их слабость для русского МСБ
- **Английский язык** — UI, документация, community-форумы. Для МСБ-оператора это барьер
- **Нет ФСТЭК-сертификата и не будет** — для регулируемых отраслей закрыто
- **Нет шаблонов отчётов под Приказ №21/№31** — нужно писать самим
- **Парсеры под российский софт отсутствуют** — 1С, Битрикс, отечественные firewall'ы нужно делать с нуля
- **Требует команду** — Wazuh + Elasticsearch + Kibana — три серьёзных продукта, нужны люди
- **Slack-канал заблокирован в РФ** — community-поддержка сложна
- **Зависимость от Elastic Stack** — JVM, прожорлив по ресурсам
- **Maxmind/VirusTotal** — внешние сервисы, не работают offline-ready

#### Тезис URSUS против Wazuh
> **«Wazuh для русского рынка: ФСТЭК-отчёты, документация на русском, парсеры под 1С и Битрикс, single-binary вместо трёх контейнеров».**

#### Где Wazuh забрать сложнее всего
Технические команды, у которых уже работает Wazuh продакшен. У них уже вложен месяц настройки, привычка к Kibana. Переходить на URSUS = повторная настройка. **Целься в тех, кто только присматривается** или столкнулся с проблемами эксплуатации Wazuh.

#### Если ЛПР спрашивает «А зачем платить, если есть Wazuh бесплатно?»
**Ответ-template:**
> «Wazuh — хороший продукт, и наш core тоже бесплатный (AGPL). Разница в трёх вещах. Первое: у нас всё на русском, включая правила и алерты — оператор не должен знать английский. Второе: парсеры под 1С, Битрикс и российские firewall'ы у нас из коробки, у Wazuh их писать самим. Третье: для прохождения проверки ФСТЭК у нас готовые отчёты по Приказу 21 — у Wazuh их нет в принципе. Платить $50/мес имеет смысл только если хотите эти три вещи без работы по их допиливанию».

Ключевой ход: **не утверждать что лучше Wazuh**, признать, что Wazuh великолепен — но локализован под другую аудиторию.

### 3.4 Против UserGate SIEM

#### Их сила
- Часть единой экосистемы UserGate (NGFW + Log Analyzer + SIEM + SOAR)
- Сертификация ФСТЭК
- Реестр российского ПО
- Прозрачная лицензия по ядрам/EPS
- Готовая интеграция со своим NGFW — сразу гора событий
- Знают рынок РФ — давно работают

#### Их слабость для МСБ
- **Vendor lock-in** — максимальная ценность только если уже стоит UserGate NGFW. Если нет — вы платите за интеграцию, которой не пользуетесь
- **Стоимость подписки** — Security Updates, SIEM Expert, кластерный режим — отдельные подписки сверху лицензии
- **ПАК-форма** — для МСБ часто избыточна, нужен железный сервер
- **Сравнительно молодой SIEM-продукт** (с 2022) — экспертизы меньше, чем у KUMA
- **Узкая поддержка open-source** — нет AGPL-core, нет GitHub-сообщества

#### Тезис URSUS против UserGate SIEM
> **«UserGate SIEM, но без привязки к UserGate NGFW и с открытым кодом».**

#### Где НЕ конкурировать
Клиенты, которые уже стоят на UserGate NGFW. Им SIEM от UserGate — естественный апсейл, и они не уйдут. **Целься в тех, у кого:**
- pfSense / OPNsense / MikroTik / Континент / ИКС / Cisco
- открытые firewall'ы (iptables, nftables)
- никакого UserGate в стеке

#### Если ЛПР спрашивает «А чем вы отличаетесь от UserGate SIEM?»
**Ответ-template:**
> «UserGate SIEM хорош, если у вас уже стоит их NGFW — тогда вы получаете готовую связку. Если NGFW другой — вы платите за SIEM, не получая бонус от экосистемы. У нас открытый core (AGPL), вы не привязаны к одному вендору, можете в любой момент мигрировать данные через OCSF-стандарт. И стоимость стартует с нуля, а не со 100 тысяч за продление».

Ключевой ход: **выделить vendor lock-in как риск**, что для МСБ — реально пугающий аргумент (учились на ESXi и Cisco).

### 3.5 Универсальные позиционные ходы

Кроме конкурентных сравнений, эти сообщения работают всегда:

#### A. «Купи свой суверенитет за 0 рублей»
Российский core под AGPL = код полностью открыт, можно форкнуть, можно проверить, никто не отключит вам обновления. **Этот аргумент сильнее работает на госкомпании и муниципалитеты**, чем на коммерцию, но в МСБ тоже резонирует у тех, кто уже обжёгся на западных вендорах в 2022.

#### B. «Скачайте и развернитесь прямо сейчас»
В отличие от KUMA/MaxPatrol/UserGate, у которых процесс «связаться с менеджером → демо → пилот → договор → лицензия → инженер на инсталляцию», у URSUS — `curl install.sh | sudo bash`. **Это конверсионное преимущество**: для МСБ психологический барьер «звонить менеджеру» огромный. Стрелка идёт от плана к одной команде.

#### C. «SIEM на одном VPS за 5 тысяч рублей в месяц»
Конкретные цифры — лучший маркетинг. KUMA-инсталляция стоит как 2 года эксплуатации URSUS на VPS у Selectel/Timeweb/Yandex.Cloud.

#### D. «Если не получится — git clone и форкайте»
Парадоксальное сообщение: «мы настолько уверены, что наш код открыт». В b2b это работает как доверие: ЛПР технический? Покажи код на GitHub. ЛПР финансовый? Покажи AGPL и скажи «вы не платите за бренд».

### 3.7 Сообщения, которых стоит избегать

Распространённые ошибки позиционирования, которые угробят продукт:

- ❌ **«Лучше чем KUMA».** Касперский — это бренд с эмоцией. Сравнение лоб-в-лоб = вас не будут слушать. Сравнивайте сегменты, а не «лучше/хуже».
- ❌ **«Wazuh + AI».** Wazuh-инженеры презрительно относятся к маркетинговому AI. AGPL-сообщество разочаруется.
- ❌ **«Российский Splunk».** Никто из МСБ не покупал Splunk и не знает, что это. Сравнение бессмысленно для аудитории.
- ❌ **«Самый дешёвый SIEM».** Дёшево = сомнительно. Лучше: «единственный SIEM с фиксированной ценой $50/мес».
- ❌ **«ИИ-powered next-gen SIEM».** Маркетинг-шум, в который никто из ЛПР больше не верит.
- ❌ **«ФСТЭК-сертифицированный».** До того, как реально сертифицирован — это ложь, которая выйдет боком при первом аудите. Используй формулировку **«готов к проверке по Приказу №21»** или **«соответствует требованиям»**.
